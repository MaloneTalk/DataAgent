# DataAgent 数据源与执行链调研报告

生成时间：2026-05-10  
范围：数据源管理、动态连接池、schema 读取、SQL 执行、安全边界

---

# 一、当前已具备的能力

## 1. 数据源 CRUD
已具备：
- 新增 / 查询 / 更新 / 删除数据源
- 按状态查询
- 按类型查询

## 2. 动态连接池
已具备：
- 按 datasourceId 创建和缓存 HikariDataSource
- 连接池移除与关闭
- 应用关闭时统一销毁

## 3. 物理 schema 读取
已具备：
- 读取表
- 读取列
- 读取主键
- 读取物理外键

## 4. SQL 执行
已具备：
- 仅允许 SELECT
- 查询超时控制
- 最大结果行数截断
- 基础正则安全校验
- 结果映射为结构化 `QueryResult`

---

# 二、当前仍缺什么

## 1. 缺数据源健康检查与连通性测试
当前没有明确的：
- 测试连接接口
- 健康检查接口
- 连接失败分级策略
- 池失效后的刷新策略

## 2. 缺密码治理与配置安全
当前风险很明显：
- `application.properties` 仍有明文 root/123
- `Datasource` 可能直接携带 password 返回
- 未见加密存储 / 脱敏处理

## 3. 缺更强 SQL 安全边界
当前主要是正则校验：
- 只能粗粒度禁止非 SELECT
- 对注释、方言、CTE、边界写法的鲁棒性有限

## 4. 缺执行审计
当前没有标准化能力记录：
- 谁执行了什么 SQL
- 用了哪个 datasource
- 扫描耗时
- 返回行数
- 是否被截断

## 5. 缺大库优化
当前还缺：
- schema 缓存
- 元数据预热
- 查询成本保护
- 慢查询标记
- 结果缓存

---

# 三、生产风险

## 1. active datasource 选第一个
这是非常大的行为风险。

## 2. 动态连接池会常驻
数据源多时会：
- 占连接
- 占内存
- 增加池管理复杂度

## 3. 200 行截断不等于低成本查询
当前只限制输出，不限制扫描成本，重查询仍可能拖垮数据库。

## 4. 缺 profile / secrets 治理
当前配置结构不适合严肃生产环境。

---

# 四、建议演进方向

## P0
1. 增加数据源连通性测试与健康检查
2. 移除明文密码，改 secrets / profile 管理
3. 显式 datasource 路由，不再靠“第一个 active”
4. 增加查询审计日志

## P1
5. schema 缓存
6. 池失效刷新与数据源测试状态管理
7. 更强 SQL 校验（AST 或更稳的 parser 方案）

## P2
8. 查询成本保护
9. 慢查询治理
10. 结果缓存与异步查询

---

# 五、结论

当前数据源与执行链已经足够支撑开发期和基础演示，但要成为“完善 DataAgent”的稳定后端，必须补：
- 安全边界
- 显式路由
- 健康检查
- 审计能力
- 大数据量执行保护
