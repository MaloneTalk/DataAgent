# 一、路线图结论

如果把当前项目放在完整 DataAgent 的建设路径上，它已经完成了：

## 已完成的基础骨架
1. **后端 Agent 主链路骨架**
2. **数据源与动态连接池骨架**
3. **表 / 列 / 关系语义层骨架**
4. **基础 SQL 执行骨架**
5. **最小 Maven + CI 基线**

但还没有完成：

## 仍待建设的产品层与平台层
1. **Agent 产品化能力**
2. **生产级数据源安全与执行治理**
3. **大数据量处理能力**
4. **前端交互层**
5. **测试与可观测性层**
6. **发布与环境治理层**

所以当前最合理的路线不是“继续零散补功能”，而是：

> 先收口现有骨架，再补平台稳定性，再做产品化增强，最后进入高级分析能力。

# 二、总体建设分层

我建议把完整 DataAgent 的能力拆成 6 层：

## Layer 1：执行基础层
负责：
- 数据源管理
- 动态连接池
- SQL 执行
- schema 读取

## Layer 2：语义理解层
负责：
- 表语义
- 列语义
- 关系语义
- 物理 + 逻辑 merge
- Agent 结构化 schema 输出

## Layer 3：Agent 产品层
负责：
- 会话
- 同步 / 流式输出
- 显式 datasource 路由
- 查询计划 / 解释
- 长查询任务

## Layer 4：交互与管理层
负责：
- 前端页面
- 数据源管理 UI
- 语义管理 UI
- 查询历史 / 收藏 / 结果导出

## Layer 5：平台稳定性层
负责：
- 测试
- 可观测性
- 配置治理
- 审计
- 限流与配额
- 查询安全边界

## Layer 6：高级分析能力层
负责：
- 指标语义
- 高级字段语义
- 表角色 / Join hint
- 血缘
- 分析辅助推理

# 三、总体分阶段路线图

# Phase 0：收口当前基线

## 目标
先把当前已经落地的骨架变成“稳定可持续迭代”的基线。

## 重点
1. 测试基线
2. 旧库 migration
3. 大数据量分页与缓存优化
4. 分页 / 搜索 / 批量操作
5. 统一实现层结构
6. 文档同步规则固化

## 结果
- 当前项目从“能跑”变成“可维护”

---

# Phase 1：数据源与执行安全升级

## 目标
让系统能更稳地跑在多数据源和更真实的环境里。

## 重点
1. 显式 datasource 路由
2. 数据源连通性测试
3. 健康检查
4. 密码脱敏 / secrets 管理
5. 查询审计日志
6. 更强 SQL 校验
7. 查询成本保护

## 结果
- 从开发演示态提升到可控执行态

---

# Phase 2：前端与交互闭环

## 目标
让 DataAgent 从后端能力变成真正可使用的产品。

## 重点
1. 前端真实工程落地
2. 聊天页 + SSE 渲染
3. 数据源管理页
4. 表 / 列 / 关系语义管理页
5. 候选表 / 字段选择器
6. 错误提示与空状态处理

## 结果
- 形成最小产品闭环

---

# Phase 3：Agent 产品化增强

## 目标
让 Agent 不只是“调用 3 个工具查表”，而是真正成为任务型数据代理。

## 重点
1. 结构化事件流
2. 查询计划展示
3. SQL 解释
4. 异步长查询任务
5. 查询取消
6. 结果续取 / 下载
7. 查询历史 / 收藏

## 结果
- Agent 从查数助手升级为数据代理产品核心

---

# Phase 4：语义层升级为分析语义层

## 目标
让 DataAgent 不只是理解结构，而是开始理解业务分析语义。

## 重点
1. 高级字段语义
   - display name
   - semantic type
   - enum schema
   - time role
   - example values
2. 表角色与过滤建议
3. 指标语义层
4. preferred relation / join hint

## 结果
- Agent 从结构 SQL 生成升级到分析 SQL 推理

---

# Phase 5：治理与平台完备化

## 目标
让系统具备长期演进、多人协作与生产运维能力。

## 重点
1. 语义层审计与版本化
2. draft / published
3. 软删除 / 回滚
4. metrics / tracing / health
5. 发布链路与回滚策略
6. 环境分层配置

## 结果
- 系统从“功能性产品”升级为“平台型产品”

---

# 四、优先级建议

## P0：现在最该做
1. 测试基线
2. 大数据量分页 / schema 缓存
3. migration / 升级方案
4. 显式 datasource 路由

## P1：紧随其后
5. 数据源安全与审计
6. 前端真实工程
7. 结构化 SSE 事件流

## P2：产品化增强
8. 异步长查询任务
9. 查询历史 / 收藏 / 导出
10. Agent 计划与解释能力

## P3：高级语义
11. 高级字段语义
12. 指标语义层
13. 分析增强语义

## P4：治理层
14. 审计 / 版本 / draft / published
15. 平台级可观测性与发布治理

---

# 五、当前最推荐的下一步

如果你要按投入产出比推进，我建议先按下面顺序做：

1. **补测试**
2. **补大数据量分页与缓存优化**
3. **补 migration / 升级方案**
4. **显式 datasource 路由**
5. **前端落地**

这样可以先把底盘打稳，再做产品闭环，最后再上高级语义与平台治理。

# 七、结论

当前项目最合理的建设路径是：

> **先稳基线 → 再补执行安全 → 再做前端闭环 → 再增强 Agent 产品能力 → 最后做高级语义与治理层。**

这样能最大化复用你当前已经完成的语义层和 Agent 主链路成果，也能避免系统在还不稳定时就过早堆高层能力。
